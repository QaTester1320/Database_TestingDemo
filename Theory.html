01. Database Foundations

What is a Database?

A structured system (DBMS) for storing and managing digital information efficiently.

Technical Example
An E-commerce DB storing 'Product_Catalog', 'Inventory_Levels', and 'User_Order_History' in relational tables.

What is DB Testing?

Verifying the invisible data layer (Schema, Triggers, Procedures) to ensure application logic is supported.

Technical Example
Validating that when a user clicks 'Buy', the DB updates the 'Stock' table and creates a 'Transaction' record simultaneously.

Need for Testing

To prevent data loss, ensure transactional safety (ACID), and protect sensitive information.

Technical Example
Ensures a partial data write (e.g., deducting money without shipping) is rolled back if the server crashes.

02. Structural Testing

Schema Testing

Validating that the DB architecture (tables/columns) matches the technical documentation.

Technical Example
Checking that 'Credit_Card_Number' is 'VARCHAR(16)' and is correctly masked/encrypted in the schema.

Trigger Testing

Verifying automated internal scripts that execute upon data modification events.

Technical Example
An 'AFTER DELETE' trigger on 'UserTable' must automatically move deleted records to 'ArchiveLogTable'.

Database Server Testing

Testing the environment settings, resource limits, and connection pools of the DB server.

Technical Example
Verify the 'max_connections' parameter allows 1,000 active sessions during a holiday sale without dropping requests.

03. Stored Logic & Business Rules

Stored Procedure Testing

Checking the internal logic of procedures, including input variables and return statuses.

Technical Example
Testing 'USP_ApplyDiscount': If 'PromoCode' = 'SAVE20', verify the 'FinalTotal' column decreases by exactly 20%.

Function Testing

Verifying scalar or table-valued functions for accuracy in data transformation.

Technical Example
A 'FN_GetAge' function must accurately calculate age in years based on the 'DateOfBirth' column and current system date.

Referential Business Logic

Checking that logic-heavy constraints correctly enforce organizational rules.

Technical Example
Ensuring the DB rejects an 'Order' if the 'AccountBalance' in the 'Customer' table is less than the 'OrderTotal'.

04. Maintenance & Regression

Data Clean-Up Testing

Validating that automated maintenance tasks purge temporary test data correctly.

Technical Example
Ensuring that a weekly job clears 'SessionLogs' older than 30 days without locking the 'Production' tables.

Database Regression

Ensuring schema changes (updates/patches) do not break existing queries or stored code.

Technical Example
After migrating to SQL Server 2022, verify all existing 'Reports' queries still return identical results.

05. Performance & Stress Testing

Load Testing

Measuring response times for standard queries under a predictable number of concurrent users.

Technical Example
Verifying that the 'ProductSearch' query executes in < 200ms when 500 users search simultaneously.

Stress Testing

Pushing the DB beyond its limits to observe recovery and breaking points.

Technical Example
Increasing user load until CPU hits 100% to check if the DB queues requests or crashes immediately.

Security Testing

Checking for vulnerabilities like SQL Injection and verifying data encryption levels.

Technical Example
Attempting to bypass login by injecting "' OR '1'='1" into the username field to verify it is sanitized.

06. Data Integrity & Validity

Data Validity Testing

Ensuring that stored data follows business constraints and format rules.

Technical Example
The 'Salary' column must reject negative values, and 'Email' must reject strings without an '@' symbol.

Data Integrity Testing

Ensuring data consistency across relationships (Primary/Foreign Keys).

Technical Example
An 'OrderID' cannot be created if the 'CustomerID' does not already exist in the 'Users' table.

07. ACID Compliance & Indexing

ACID Compliance

Testing the 4 pillars of transactions: Atomicity, Consistency, Isolation, Durability.

Technical Example
Atomicity: Verify that if a payment fails, the 'Order' status is NOT updated to 'Paid'. It's all or nothing.

Index Testing

Validating that indexes exist for frequently searched columns to maintain high speed.

Technical Example
Querying 10M 'Transactions' by 'Date': Verify it takes 50ms with an Index vs 10s with a Full Table Scan.
